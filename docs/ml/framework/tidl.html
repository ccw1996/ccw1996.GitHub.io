#   向量数据类型
    向量数据类型名由标量数据类型名及元素个数组成。数据类型名表示元素所占内存空间以及是否有符号，元素个数为 2, 3, 4, 8, 16, 32, 或 64。

    | 元素类型名 | 元素类型说明 | 向量数据类型 |
    |---|---|---|
    | char | 8 比特有符号整数 | char2, char3, char4, char8, char16, char32, char64 |
    | uchar | 8 比特无符号整数 | uchar2, uchar3, uchar4, uchar8, uchar16, uchar32, uchar64 |
    | short | 16 比特有符号整数 | short2, short3, short4, short8, short16, short32 |
    | ushort | 16 比特无符号整数 | ushort2, ushort3, ushort4, ushort8, ushort16, ushort32 |
    | int | 32 比特有符号整数 | int2, int3, int4, int8, int16 |
    | uint | 32 比特无符号整数 | uint2, uint3, uint4, uint8, uint16 |
    | long | 64 比特有符号整数 | long2, long3, long4, long8 |
    | ulong | 64 比特无符号整数 | ulong2, ulong3, ulong4, ulong8 |
    | float | 32 比特单精度浮点数 | float2, float3, float4, float8, float16 |
    | double | 64 比特双精度浮点数 | double2, double3, double4, double8 |
    | cchar | 实部和虚部均为 8 比特有符号整数 | cchar2, cchar4, cchar8, cchar16, cchar32 |
    | cshort | 实部和虚部均为 16 比特有符号整数 | cshort2, cshort4, cshort8, cshort16 |
    | cint | 实部和虚部均为 32 比特有符号整数 | cint2, cint4, cint8 |
    | clong | 实部和虚部均为 64 比特有符号整数 | clong2, clong4 |
    | cfloat | 实部和虚部均为 32 比特单精度浮点数 | cfloat2, cfloat4, cfloat8 |
    | cdouble | 实部和虚部均为 64 比特双精度浮点数 | cdouble2, cdouble4 |
    如果占用的内存大小超过 64 比特，则在内存中按照 8 字节对齐 (即占用空间的首地址能被 8 整除)；如果占用内存大小不超过 64 比特，则在内存中按照其占据的字节数对齐 (例如 _short2 _占据 4 字节，在内存中按照 4 字节对齐)。
    ## 多维数据的存储及遍历
        一次读取多个数据以配合 SIMD (例如一次读取 64 字节)，这些数据在内存中连续。
        一次读取的数据不“跨行”。这里的“行”定义为上图中的橙色矩形框，即一行包含 _N0_ 个数据。由于要求读取的数据在内存中连续，如果存在用于字节对齐的 _Buf_ (上图中的灰色矩形框)，那么跨行读取会包含脏数据。
        限制了跨行读取后，每行数据的最后一次读取，末尾可能包含多个无效数据，且需要特别注意以防止数组越界行为。
    ## 如何填满 512 比特数据
        流引擎从内存中获取数据。流引擎获取的数据不一定正好为 512 比特。例如 _ELETYPE_ 为 _**SE_ELETYPE_8BIT _、 _VECLEN _为 **_SE_VECLEN_2ELEMS_，则流引擎一次只获取 16 比特的数据。
        流引擎将获取的数据填入特定的只读寄存器。尽管获取的数据不一定为 512 比特，但该只读寄存器会提供 512 比特的数据。在上述只获取 16 比特数据的例子中，这 16 比特的数据会写入只读寄存器的低 16 位，剩下的 496 位填 0。(注，这里不考虑越界检测问题。)
        用户解析该只读寄存器中的数据。在调用 _**SEn()_ 或 **_SEnADV()_ (n = 0,1) 接口时，传入的类型参数 _type_ 表示要如何解析只读寄存器中的数据。如果 _type_ 类型占据的内存空间小于 512 比特，则只会解析只读寄存器中低位的数据。例如 _type_ 为 _char32_，则只解析只读寄存器的低 256 比特的数据。
        ### ELETYPE & VECLEN
          
        这两个是最基础且必须的配置，仅考虑这两个配置。若希望以此获取 512 比特的数据，需要 _ELETYPE _× _VECLEN _= 512 比特。例如 _ELETYPE _为 _**SE_ELETYPE_8BIT_，则 _VECLEN _为 **_SE_VECLEN_64ELEMS_。  
        ### ELETYPE & VECLEN & PROMOTE
          
        *PROMOTE _的功能为对读取的数据进行扩容，需要 _ELETYPE _× _VECLEN _× _PROMOTE *= 512 比特。  
          
        例如 _ELETYPE _为 _**SE_ELETYPE_8BIT_，即每个数据大小为 8 比特。如果 _PROMOTE _为 ***SE_PROMOTE_2X_ZEROEXT_，则读取后每个数据的大小为 16 比特。此时 _VECLEN _需为 *__SE_VECLEN_32ELEMS_。  
          
        扩容功能解决数据溢出的问题。例如期望进行 _uchar64 _数据之间的相乘，根据 C7x DSP 相关指令，结果仍然为 _uchar64 _数据，很可能溢出，将数据扩容成 _ushort32 _可解决该问题。当然，扩容后数据读取次数及乘法运算次数均是原来的 2 倍。  
        ### ELETYPE & VECLEN & ELEDUP
          
        要填满 512 比特数据仍要求 _ELETYPE _× _VECLEN _= 512 比特，不过这些数据只来自 _VECLEN _/ _ELEDUP _个“原始”数据。例如 _ELETYPE _为 _**SE_ELETYPE_8BIT_， _VECLEN _为 ***SE_VECLEN_64ELEMS_，_ELEDUP _为 *__SE_ELEDUP_4X_，那么流引擎一次获取 16 个数据并对每个数据复制成 4 份。  
        ### ELETYPE & VECLEN & GRPDUP
          
        *GRPDUP _设置为 ***SE_GRPDUP_ON _以开启“群复制”功能。群复制的复制次数是根据其它成员自动推断的，例如 _ELETYPE _为 ***SE_ELETYPE_8BIT_， _VECLEN _为 ***SE_VECLEN_64ELEMS_，则不会进行群复制，因为已经填满了 512 比特的数据。又如 _ELETYPE _为 ***SE_ELETYPE_8BIT_， _VECLEN _为 *__SE_VECLEN_16ELEMS_，则会对获取的 16 个数据**整体**复制 4 次，从而填满 512 比特的数据**。**

    ## 越界检测
        ### 比较   *ICNT0 * 及   *i0*
          
        在某一次读取中，假设会读取 _N_ 个数据，则这 _N_ 个数据的索引分别为 (这里省略 _i1 _~* i5_ 对索引值的计算)：_i0_，_i0*+1, ... _i0_+*N*-1。假如第 _k_ (<*N*) 个数据的索引正好满足 _i0_+*k*-1 = _ICNT0_-1，则剩下的 _N_-_k_ 个数据会使用 0 来代替。  
    C7x DSP 二级缓存总大小为 512 K，划分为 SRAM 及 Cache 两部分。其中 Cache 在 _vision_apps_ 中作为 Scratch Memory (不太清楚如何翻译，暂存器？) 来使用，也是本文的介绍对象。Scratch Memory 由所有任务共用 (C7x DSP 运行 RTOS 系统)，即两个任务可能会对 Scratch Memory 内同一地址进行读写。目前没有发现切换任务时会保存 Scratch Memory 数据的机制，如果担心 Scratch Memory 中的数据被污染，可以调用相关接口暂时屏蔽中断以阻止任务调度。
    要使用 Scratch Memory 仍然需要调用相关接口申请内存，如果使用 TIOVX，则调用 _tivxMemAlloc()_ 接口。如果深究 _tivxMemAlloc()_ 的源码，就会发现其内部调用 _appMemAlloc()_ 函数完成内存申请——即需要使用者自行提供 _appMemAlloc()_ 函数的实现，当然 _vision_apps_ 已经替使用者提供了实现 Demo。
    *vision_apps_ 按照线性内存分配的方式管理 Scratch Memory。假如 Scratch Memory 的首地址为 _head_，已经分配的内存为 _m_，则下一次分配的内存首地址 _addr_next _满足 _addr_next _≥_ head *+* m _且 _addr_next _满足字节对齐 (如满足 8 字节对齐，即 _addr_next *% 8 == 0)。
    ### N 维数据传输配置结构体
    _vision_apps/apps/basic_demos/app_c7x_kernel/_ 中提供了一个示例，实现两个图像对应像素的相加操作。该示例实现为 3 段流水线模型，每段的功能分别为：
    分别使用 DMA 从外部内存获取两个图像的数据，并搬运到 Scratch Memory 中。
    从 Scratch Memory 中获取图像数据并进行对应像素的相加操作，计算结果存入 Scratch Memory 中。
    使用 DMA 将计算结果从 Scratch Memory 搬运到外部内存中。

## **C7x DSP 主要模块**
    矩阵乘法加速器(MMA)作为C71x CorePac CPU的特殊功能单元
    L1程序内存控制器(PMC)具有32KB的L1内存
    L1数据内存控制器(DMC)具有48KB L1D内存，可配置为缓存和/或SRAM
    L2统一内存控制器(UMC)， 512KB L2内存，可配置为缓存和/或SRAM
双数据路径 A-64位标量 和 B-512位矢量
A面可在同一时钟周期内并发多种标量运算和内存搬运
B面 可在同一时钟周期内并发多种矢量（定点、浮点）运算和内存搬运

winograd

一维卷积
    输出为 m，卷积核长为 r，对应的乘法计算次数为 m+r-1 次
二维卷积
    如果输出维度是 m x n，卷积核维度为 r x s，则需要的乘法次数是 (m+r-1) x (n+s-1)
winograd 算法通过减少乘法次数来实现提速，但是加法的数量会相应增加，同时需要额外的转换计算以及存储转换矩阵
